TL;DR

  • Multi-fidelity: the Kennedy-O'Hagan model needs only 15–30 HF points to correct a 200-point LF surrogate. Check $\rho$: if $\rho \ll 0.8$, the LF model has systematic bias — fix the physics first.
  • Physics constraints are non-optional. A surrogate that violates conservation laws is not just inaccurate — it is dangerous if used for control or certification.
  • Three enforcement strategies in order of flexibility: (1) architecture constraints (zero cost, hard guarantees), (2) physics loss/PINN (flexible, PDE-based), (3) grey-box (partial physics + ML residual, the default).
  • PINNs work well for forward problems with known PDEs and limited data. They fail for high-Re turbulence, discontinuities (shocks), and hard geometry constraints — know the failure modes before committing.
  • SINDy discovers governing equations from data: output is interpretable differential equations, not a black box. The right tool when you suspect the governing equation structure but don't know the coefficients.
  • POD-ROM reduces full-field CFD/FEM to a low-dimensional linear system: 99% of the variance explained by the first 20–50 modes in most engineering flows.
  • Always apply dimensional analysis (Buckingham Pi) before feature selection — it reduces dimension count and guarantees scale-invariant models.

1. The Multi-Fidelity Problem: Cost vs. Accuracy

Engineering workflows generate data at multiple fidelity levels, each with a different cost-accuracy trade-off. The fundamental multi-fidelity goal: leverage abundant cheap data to reduce the number of expensive high-fidelity evaluations needed to reach a given accuracy target.

2. Kennedy-O'Hagan Auto-Regressive Model

The KO model (Kennedy & O'Hagan, 2000) links two fidelity levels with a multiplicative scale factor $\rho$ and an additive discrepancy GP $\delta$:

$$y_{HF}(\mathbf{x}) = \rho\,y_{LF}(\mathbf{x}) + \delta(\mathbf{x}), \qquad \delta \sim \mathcal{GP}(0, k_\delta(\cdot,\cdot))$$

The KO model is fitted in two stages:

  1. Fit a GP to LF data: $\hat{y}_{LF}(\mathbf{x}) = \mu_{LF}(\mathbf{x})$
  2. Estimate $\rho$ by OLS on HF data at the LF prediction: $\hat{\rho} = \arg\min_\rho \|y_{HF} - \rho\hat{y}_{LF}\|^2$
  3. Compute discrepancy: $\hat{\delta}(\mathbf{x}) = y_{HF}(\mathbf{x}) - \hat{\rho}\hat{y}_{LF}(\mathbf{x})$
  4. Fit a second GP to the discrepancy residuals
import gpflow, numpy as np
from sklearn.preprocessing import StandardScaler

def kennedy_ohagan(X_lf, y_lf, X_hf, y_hf):
    """Two-level Kennedy-O'Hagan multi-fidelity GP."""
    scaler = StandardScaler()
    Xl = scaler.fit_transform(X_lf)
    Xh = scaler.transform(X_hf)

    # 1. LF GP
    k_lf = gpflow.kernels.Matern52(lengthscales=np.ones(X_lf.shape[1]))
    gp_lf = gpflow.models.GPR((Xl, y_lf[:,None]), kernel=k_lf, noise_variance=1e-5)
    gpflow.optimizers.Scipy().minimize(gp_lf.training_loss, gp_lf.trainable_variables)

    # 2. LF predictions at HF sites → compute rho and discrepancy
    mu_lf, _ = gp_lf.predict_y(Xh)
    mu_lf = mu_lf.numpy().ravel()
    rho = float(np.linalg.lstsq(mu_lf[:,None], y_hf[:,None], rcond=None)[0][0,0])
    delta = y_hf - rho * mu_lf
    print(f"rho = {rho:.4f}  (1.0 = LF perfectly calibrated)")

    # 3. Discrepancy GP (augmented with LF prediction as input)
    Xh_aug = np.hstack([Xh, mu_lf[:,None]])
    k_d = gpflow.kernels.Matern52(lengthscales=np.ones(Xh_aug.shape[1]))
    gp_d = gpflow.models.GPR((Xh_aug, delta[:,None]), kernel=k_d, noise_variance=1e-5)
    gpflow.optimizers.Scipy().minimize(gp_d.training_loss, gp_d.trainable_variables)
    return gp_lf, gp_d, rho, scaler

def mf_predict(gp_lf, gp_d, rho, scaler, X_new):
    Xn = scaler.transform(X_new)
    mu_lf, _ = gp_lf.predict_y(Xn)
    Xn_aug = np.hstack([Xn, mu_lf.numpy()])
    mu_d, var_d = gp_d.predict_y(Xn_aug)
    mu_hf = rho * mu_lf.numpy().ravel() + mu_d.numpy().ravel()
    sigma  = np.sqrt(var_d.numpy().ravel())
    return mu_hf, sigma

3. Budget Allocation and Multi-Level Extensions

The optimal allocation minimises integrated posterior variance subject to total cost budget $\mathcal{B}$:

$$\min_{n_{LF}, n_{HF}} \int_\mathcal{X} \sigma^{*2}_{MF}(\mathbf{x})\,d\mathbf{x} \quad \text{s.t.} \quad c_{LF} n_{LF} + c_{HF} n_{HF} \leq \mathcal{B}$$
Budget Allocation Rules
  • If $c_{HF}/c_{LF} > 10$: invest 80% of budget in LF, 20% in HF.
  • Minimum HF points: $\geq 5(d+1)$ for the discrepancy GP to be identifiable in $d$ dimensions. For $d=5$: at least 30 HF points.
  • LF density: $\geq 10\times$ HF count. More LF is almost always beneficial since it is cheap.
  • Stop collecting HF when $\max_\mathbf{x} \sigma^*_{MF}(\mathbf{x}) < \epsilon_{target}$. Additional HF beyond this threshold delivers negligible improvement.

3.1 Three-Level KO Extension

Extending KO to three levels (L1, L2, L3) nests two auto-regressive corrections:

$$y_{L2}(\mathbf{x}) = \rho_1 y_{L1}(\mathbf{x}) + \delta_1(\mathbf{x})$$ $$y_{L3}(\mathbf{x}) = \rho_2 y_{L2}(\mathbf{x}) + \delta_2(\mathbf{x})$$

In practice, the three-level model rarely outperforms two-level by more than 10–15% in RMSE — the marginal gain from adding a third fidelity level is small unless the cost ratios between adjacent levels are very different. Prefer two-level KO unless the problem specifically motivates three levels.

4. Why Physics Constraints Are Non-Negotiable

A data-driven surrogate trained without physics constraints can violate:

  • Conservation laws: negative density in CFD, energy not conserved in thermal model
  • Thermodynamic consistency: entropy decrease in an irreversible process (2nd law violation)
  • Monotone relationships: compressor pressure ratio decreasing with mass flow at fixed speed (physically impossible at non-surge conditions)
  • Symmetry: axisymmetric geometry producing asymmetric flow field predictions
  • Dimensional consistency: predicting stress in Pascals that depends on temperature in a way that's unit-inconsistent

In engineering, these violations are not just inaccuracies — they are dangerous. A virtual sensor that predicts negative turbine inlet temperature drives the control system into an incorrect safety action. A surrogate-assisted optimiser that exploits a non-physical response surface finds infeasible designs.

5. Architecture-Based Physics Constraints

5.1 Monotone Networks (Input Convex Neural Networks)

To guarantee $\partial f/\partial x_j \geq 0$ for specified input $x_j$: constrain all weights in layers connected to $x_j$ to be non-negative (use softplus or absolute value activation on weights). For a partially monotone network:

import torch
import torch.nn as nn

class MonotoneLayer(nn.Linear):
    """Linear layer with guaranteed non-negative weights for monotone inputs."""
    def forward(self, x):
        return torch.nn.functional.linear(x, self.weight.abs(), self.bias)

class PartialMonotoneNet(nn.Module):
    """
    Surrogate with monotone constraint on specified input dimensions.
    mono_dims: list of input indices that must have non-negative partial derivative.
    """
    def __init__(self, d_in, d_out, hidden, mono_dims):
        super().__init__()
        self.mono_dims = mono_dims
        d_free = d_in - len(mono_dims)

        # Free path (unconstrained inputs)
        self.free_path = nn.Sequential(
            nn.Linear(d_free, hidden), nn.GELU()
        )
        # Monotone path (constrained inputs — non-negative weights)
        self.mono_path = nn.Sequential(
            MonotoneLayer(len(mono_dims), hidden), nn.Softplus()
        )
        # Combined output
        self.out = MonotoneLayer(hidden * 2, d_out)  # output must also be monotone in hidden

    def forward(self, x):
        free_x = x[:, [i for i in range(x.shape[1]) if i not in self.mono_dims]]
        mono_x = x[:, self.mono_dims]
        h = torch.cat([self.free_path(free_x), self.mono_path(mono_x)], dim=1)
        return self.out(h)

5.2 Input Convex Neural Networks (ICNN)

ICNNs (Amos et al., 2017) guarantee that the output is convex in the inputs — required for thermodynamic potentials (Gibbs free energy, Helmholtz free energy) and for correct behaviour in optimisation (convex landscape ensures no spurious local minima in the surrogate).

5.3 Equivariant Networks for Symmetric Physics

If the physical system is symmetric under rotation, reflection, or permutation, the surrogate should respect the same symmetry. Equivariant networks encode symmetry in the layer architecture — the output transforms predictably under the same group action as the input. For 3D engineering geometries: $E(3)$-equivariant networks (SchNet, DimeNet, NequIP) are the standard for molecular/structural ML. For turbomachinery with $k$-fold rotational symmetry: $Z_k$-equivariant models.

6. Physics-Informed Neural Networks (PINNs)

PINNs (Raissi et al., 2019) embed PDE constraints directly into the training loss by evaluating the PDE residual at collocation points using automatic differentiation:

$$\mathcal{L}_{total} = \underbrace{\mathcal{L}_{data}}_{\text{fit observations}} + \lambda\underbrace{\mathcal{L}_{PDE}}_{\text{PDE residual}} + \mu\underbrace{\mathcal{L}_{BC}}_{\text{boundary conditions}}$$ $$\mathcal{L}_{PDE} = \frac{1}{N_c}\sum_{i=1}^{N_c} \left\|\mathcal{N}[\hat{u}](\mathbf{x}_i^c, t_i^c) - f(\mathbf{x}_i^c, t_i^c)\right\|^2$$

where $\mathcal{N}[\hat{u}]$ is the differential operator applied to the NN prediction via autograd, and $\mathbf{x}_i^c$ are collocation points (no label required — only the PDE needs to be satisfied there).

import torch
import torch.nn as nn

class PINN_HeatConduction(nn.Module):
    """
    PINN for 1D transient heat conduction: rho*cp * dT/dt = k * d2T/dx2 + q
    """
    def __init__(self, alpha=1.0):  # thermal diffusivity
        super().__init__()
        self.alpha = alpha
        self.net = nn.Sequential(
            nn.Linear(2, 64), nn.Tanh(),  # input: (x, t)
            nn.Linear(64, 64), nn.Tanh(),
            nn.Linear(64, 64), nn.Tanh(),
            nn.Linear(64, 1)
        )

    def forward(self, x, t):
        return self.net(torch.cat([x, t], dim=1))

    def pde_residual(self, x, t):
        """Compute heat equation residual dT/dt - alpha * d2T/dx2"""
        x.requires_grad_(True); t.requires_grad_(True)
        T = self(x, t)
        T_t  = torch.autograd.grad(T.sum(), t, create_graph=True)[0]
        T_x  = torch.autograd.grad(T.sum(), x, create_graph=True)[0]
        T_xx = torch.autograd.grad(T_x.sum(), x, create_graph=True)[0]
        return T_t - self.alpha * T_xx

def train_pinn(model, X_data, T_data, X_colloc, n_epochs=10000, lam=1.0):
    opt = torch.optim.Adam(model.parameters(), lr=1e-3)
    for epoch in range(n_epochs):
        opt.zero_grad()
        # Data loss
        T_pred = model(X_data[:, :1], X_data[:, 1:])
        loss_data = ((T_pred - T_data)**2).mean()
        # PDE residual loss at collocation points
        residual = model.pde_residual(X_colloc[:, :1], X_colloc[:, 1:])
        loss_pde = (residual**2).mean()
        loss = loss_data + lam * loss_pde
        loss.backward()
        opt.step()
        if epoch % 1000 == 0:
            print(f"Epoch {epoch}: data={loss_data:.2e}, PDE={loss_pde:.2e}")

6.1 PINN Failure Modes and When NOT to Use Them

Pitfall — PINN Failure Modes
  • High Reynolds number turbulence: PINNs cannot resolve turbulent boundary layers without spatial resolution equivalent to DNS — the NN cannot represent the solution. Use POD-ROM or data-driven turbulence closures instead.
  • Discontinuities / shocks: the smoothness constraint of the NN activation functions fights against sharp gradients. Standard PINNs fail for compressible flow with shocks; use physics-constrained FNO or adaptive PINNs.
  • Slow convergence: PINN training is slower and less reliable than supervised learning. The $\lambda$ weight balancing between data and PDE loss is problem-specific and requires tuning. Pathological loss landscapes (gradient imbalance) are common.
  • Stiff ODEs/PDEs: PINNs struggle with stiff systems (vastly different time scales). Use implicit time-stepping or separate fast/slow scale treatment.
  • Large domains: PINNs scale poorly with domain size and dimensionality. For 3D CFD, they are not competitive with FNO or POD-ROM for similar computational cost.

6.2 When PINNs Are Effective

  • Very sparse observation data in a domain where the PDE is known (inverse problems)
  • Forward problems for smooth, low-Re flows with simple geometry
  • Parameter identification from indirect measurements (e.g., estimating thermal conductivity from surface temperature)
  • Initial/boundary value problems where you need a continuous, differentiable solution for post-processing

7. Physics-Informed Gaussian Processes

Physics constraints can be incorporated into GPs by treating derivative observations, inequality constraints, or entire PDE solutions as observations or prior structure.

7.1 Derivative Observations

For a GP $f \sim \mathcal{GP}(\mu, k)$, derivative processes $\partial f/\partial x_j$ are also GPs by linearity of differentiation. Including known derivative observations (e.g., from finite-difference approximations of CFD results, or from physical gradients) tightens the posterior and enforces gradient structure:

$$\text{Cov}\!\left(\frac{\partial f}{\partial x_j}(\mathbf{x}), f(\mathbf{x}')\right) = \frac{\partial k(\mathbf{x}, \mathbf{x}')}{\partial x_j}$$ $$\text{Cov}\!\left(\frac{\partial f}{\partial x_j}(\mathbf{x}), \frac{\partial f}{\partial x_j'}(\mathbf{x}')\right) = \frac{\partial^2 k(\mathbf{x}, \mathbf{x}')}{\partial x_j \partial x_j'}$$

7.2 Monotone GP

To enforce that $f$ is monotonically increasing in $x_j$ everywhere, add virtual observations $\partial f/\partial x_j(\mathbf{x}^v) \geq 0$ at a grid of virtual points $\mathbf{x}^v$ using a truncated Gaussian likelihood. The EP (Expectation Propagation) or Laplace approximation handles these inequality constraints efficiently (Riihimäki & Vehtari, 2010).

7.3 Physical Kernel Design

The kernel encodes prior beliefs about function smoothness and periodicity. For engineering surrogates with known physical structure:

  • Periodic kernel $k_{per}(\mathbf{x}, \mathbf{x}') = \sigma_f^2 \exp(-2\sin^2(\pi\|\mathbf{x}-\mathbf{x}'\|/p)/\ell^2)$: for signals known to be periodic (rotating machinery, seasonal process variables)
  • Additive kernel $k(\mathbf{x},\mathbf{x}') = \sum_j k_j(x_j, x_j')$: when inputs affect the output independently (separable physics)
  • Product kernel: for multiplicative interaction effects
  • Linear + RBF: for outputs that are approximately linear with a nonlinear correction

8. Grey-Box ODE Systems

The grey-box ODE approach combines a known (or partially known) physics model (ODE/PDE) with a data-driven correction term. It is the default architecture for regulated engineering domains because it provides physical bounds on extrapolation and reduces data requirements by 40–60% vs. pure black-box.

$$\dot{\mathbf{x}}(t) = \underbrace{f_{phys}(\mathbf{x}(t), \mathbf{u}(t), \boldsymbol{\theta})}_{\text{known physics}} + \underbrace{g_{ML}(\mathbf{x}(t), \mathbf{u}(t), t)}_{\text{learned correction}}$$ $$\text{or in algebraic form: } y(\mathbf{x}) = \underbrace{y_{phys}(\mathbf{x})}_{\text{0D physics model}} + \underbrace{\delta(\mathbf{x})}_{\text{ML residual (GP or NN)}}$$

The algebraic form is the default for steady-state engineering maps (compressor efficiency, NOx vs. ECU inputs). The ODE form is used when the dynamics of the system are known but the forcing terms are uncertain (e.g., thermal model with unknown heat generation from side reactions).

import torch
import numpy as np
from torchdiffeq import odeint

class GreyBoxODE(torch.nn.Module):
    """
    Grey-box ODE: known physics + learned NN correction.
    Physics: Newton's cooling law  dT/dt = -k*(T - T_amb)
    Correction: NN captures unmodelled heat sources/sinks.
    """
    def __init__(self, k_cool=0.05, T_amb=25.0):
        super().__init__()
        self.k_cool = k_cool
        self.T_amb  = T_amb
        self.correction = torch.nn.Sequential(
            torch.nn.Linear(2, 32), torch.nn.Tanh(),
            torch.nn.Linear(32, 32), torch.nn.Tanh(),
            torch.nn.Linear(32, 1)
        )

    def forward(self, t, state):
        T = state
        # Physics term (Newton's cooling)
        physics = -self.k_cool * (T - self.T_amb)
        # Data-driven correction (inputs: T, t)
        xt = torch.cat([T.unsqueeze(-1), t.expand_as(T).unsqueeze(-1)], dim=-1)
        delta = self.correction(xt).squeeze(-1)
        return physics + delta

# Training: minimise MSE on measured temperature trajectory
model = GreyBoxODE()
t_span = torch.linspace(0, 300, 60)      # 5 minutes, 5s intervals
T0     = torch.tensor([90.0])            # initial temperature

def train_greybox(model, t_span, T_measured, n_epochs=2000):
    opt = torch.optim.Adam(model.parameters(), lr=1e-3)
    for epoch in range(n_epochs):
        opt.zero_grad()
        T_pred = odeint(model, T0, t_span, method='dopri5')[:,0]
        loss   = ((T_pred - T_measured)**2).mean()
        loss.backward(); opt.step()
    return model

9. SINDy: Sparse Identification of Nonlinear Dynamics

SINDy (Brunton, Proctor & Kutz, 2016) discovers the governing equations of a dynamical system from data by assuming that the right-hand side of the ODE lives in a sparse linear combination of a library of candidate functions:

$$\dot{\mathbf{x}} = \boldsymbol{\Xi}\boldsymbol{\Theta}(\mathbf{x})^\top$$ $$\text{where } \boldsymbol{\Theta}(\mathbf{x}) = [1,\, x_1,\, x_2,\, x_1^2,\, x_1 x_2,\, x_2^2,\, \sin(x_1),\, \ldots]$$

$\boldsymbol{\Xi}$ is the sparse coefficient matrix. SINDy solves for $\boldsymbol{\Xi}$ by sparse regression (LASSO, Sequential Thresholded Least Squares — STLS) on the regression problem $\dot{\mathbf{X}} = \boldsymbol{\Theta}(\mathbf{X})\boldsymbol{\Xi}^\top$.

import pysindy as ps
import numpy as np

# Time-series data of state variables
t = np.linspace(0, 20, 1000)
x = measurement_data  # (1000, n_states)

# Define candidate function library
library = ps.PolynomialLibrary(degree=3)  # 1, x, x^2, x^3, x*y, ...
# or combine with trigonometric terms:
# library = ps.GeneralizedLibrary([ps.PolynomialLibrary(2), ps.FourierLibrary()])

# Fit SINDy
model = ps.SINDy(
    optimizer=ps.STLSQ(threshold=0.1),   # STLS: enforces sparsity
    feature_library=library,
    differentiation_method=ps.FiniteDifference()  # or ps.SINDyDerivative
)
model.fit(x, t=t)
model.print()  # prints discovered equations in human-readable form

# Example output:
# dx0/dt = -0.997 x0 + 2.001 x0 x1
# dx1/dt = 0.998 x1 - 1.002 x0 x1
When SINDy is the Right Tool
  • You suspect the governing equation structure (ODE) but don't know the coefficients or nonlinear terms
  • You want an interpretable, low-complexity model — SINDy output is a set of equations, not a black box
  • The system is low-dimensional (n_states ≤ 10) and the data is clean enough for numerical differentiation
  • You need to extrapolate: discovered equations can be integrated forward in time using standard ODE solvers, unlike NNs which are unreliable far from training data
When SINDy fails: high noise (differentiation amplifies noise severely), high-dimensional systems ($d > 20$ makes the library intractably large), and when the underlying dynamics are not sparse in any reasonable function library.

10. POD-ROM: Proper Orthogonal Decomposition

Proper Orthogonal Decomposition (POD) — also known as Principal Component Analysis in the data science community, or Karhunen-Loève expansion in probability theory — finds the optimal low-dimensional basis for representing a high-dimensional field:

$$\mathbf{u}(\mathbf{x}, t) \approx \sum_{k=1}^r a_k(t)\,\boldsymbol{\phi}_k(\mathbf{x}), \qquad \boldsymbol{\Phi} = [\boldsymbol{\phi}_1, \ldots, \boldsymbol{\phi}_r] \text{ from SVD of snapshot matrix } \mathbf{U}$$ $$\mathbf{U} = \boldsymbol{\Phi}\boldsymbol{\Sigma}\mathbf{V}^\top, \qquad r = \min\{k : \sum_{i=1}^k \sigma_i^2 / \sum_{i=1}^n \sigma_i^2 \geq 0.99\}$$

The first $r$ POD modes capture 99% of the variance in the ensemble of CFD/FEM snapshots. In most engineering flows, $r = 20$–50 modes suffice, reducing the state dimension from millions (mesh nodes) to tens.

import numpy as np

def pod_rom(snapshot_matrix, energy_threshold=0.99):
    """
    Compute POD modes from a snapshot matrix.
    snapshot_matrix: (n_dof, n_snapshots) — columns are CFD field solutions
    Returns: modes (n_dof, r), singular values (r,), energy fractions (r,)
    """
    U, sigma, Vt = np.linalg.svd(snapshot_matrix, full_matrices=False)

    # Select r modes to capture energy_threshold of total energy
    energy = np.cumsum(sigma**2) / np.sum(sigma**2)
    r = np.searchsorted(energy, energy_threshold) + 1
    print(f"POD: {r} modes capture {energy[r-1]*100:.1f}% of variance")

    modes = U[:, :r]             # POD basis (n_dof, r)
    coeffs = sigma[:r, None] * Vt[:r, :]  # temporal coefficients (r, n_snapshots)
    return modes, sigma[:r], energy[:r], coeffs

# Use: Galerkin projection (reduced-order model)
# Substitute u = Phi @ a into PDE, multiply by Phi^T → r ODE for a(t)
# This is the standard Galerkin ROM — see Rowley & Dawson (2017)

10.1 POD-ROM for Parametric Studies

For surrogate-accelerated design optimisation, the POD-Galerkin ROM is too intrusive (requires access to the CFD solver). A non-intrusive alternative:

  1. Run $n$ CFD snapshots at different design parameters $\boldsymbol{\mu}_i$.
  2. Compute POD modes $\boldsymbol{\Phi}$ from the snapshot ensemble.
  3. Project each snapshot onto the POD basis: $\mathbf{a}_i = \boldsymbol{\Phi}^\top \mathbf{u}_i$.
  4. Train a surrogate (GP or NN) from $\boldsymbol{\mu}_i \to \mathbf{a}_i$ (cheap: $r$ outputs instead of millions).
  5. Predict full field for new parameters: $\hat{\mathbf{u}}(\boldsymbol{\mu}^*) = \boldsymbol{\Phi}\,\hat{\mathbf{a}}(\boldsymbol{\mu}^*)$.

This is the standard POD-surrogate pipeline used in industrial MDO (multidisciplinary design optimisation). For $n=100$ snapshots, $r=30$ modes, and a 10-dimensional design space, the GP surrogate for the coefficients requires $100 \times 30$ training points — tractable with any standard GP library.

11. Dimensional Analysis and Buckingham Pi for Feature Engineering

Before engineering any features or building any surrogate, apply dimensional analysis. It reduces the dimensionality of the problem, reveals physically meaningful input combinations, and guarantees that the surrogate is scale-invariant (same prediction in SI and imperial units).

11.1 Buckingham Pi Theorem

If a physical system has $n$ dimensional variables governed by $k$ independent base dimensions (M, L, T, θ, …), the system can be described by $n - k$ dimensionless groups $\Pi_i$:

$$f(q_1, q_2, \ldots, q_n) = 0 \implies F(\Pi_1, \Pi_2, \ldots, \Pi_{n-k}) = 0$$
Example — Compressor Surrogate: 7 Variables → 3 Pi Groups
Variables: $\dot{m}$ [kg/s], $N$ [rpm], $T_1$ [K], $p_1$ [Pa], $D$ [m], $\eta_{is}$ [-], $\Pi_R$ [-]
Base dimensions: M, L, T, θ — so $k = 4$, $n-k = 3$ Pi groups
Standard dimensionless groups:
  • Corrected mass flow: $\Pi_1 = \dot{m}\sqrt{T_1} / (p_1 D^2)$
  • Corrected speed: $\Pi_2 = N D / \sqrt{T_1}$ (proportional to blade Mach number)
  • Pressure ratio: $\Pi_3 = \Pi_R = p_2/p_1$
Result: $\eta_{is} = f(\Pi_1, \Pi_2, \Pi_3)$ — the surrogate has 3 inputs instead of 7. The model trained on standard-day data automatically predicts performance at any altitude and ambient temperature — the Pi groups collapse all conditions. 
import numpy as np

def compute_compressor_pi_groups(m_dot, N_rpm, T1_K, p1_Pa, D_m):
    """
    Compute corrected mass flow and corrected speed for compressor map.
    T_ref=288.15K, p_ref=101325Pa (ISA sea level standard day)
    """
    T_ref, p_ref = 288.15, 101325.0
    theta = T1_K / T_ref      # temperature ratio
    delta = p1_Pa / p_ref     # pressure ratio to sea level

    m_corr = m_dot * np.sqrt(theta) / delta   # corrected mass flow [kg/s]
    N_corr = N_rpm / np.sqrt(theta)            # corrected speed [rpm]
    return m_corr, N_corr

# Now build surrogate on (m_corr, N_corr, PR) → eta_is
# instead of (m_dot, N_rpm, T1, p1, D) → eta_is
Dimensional Analysis Protocol Before Building Any Surrogate
  1. List all input and output variables with their physical dimensions.
  2. Count the number of independent base dimensions $k$ (M, L, T, θ, I, N, J — usually 3–5).
  3. Form $n-k$ dimensionless Pi groups using the repeating variables method or an SVD-based algorithm.
  4. Use the Pi groups as surrogate inputs instead of the original dimensional variables.
  5. Validate that the surrogate trained on one operating condition (T1=300K) correctly predicts another (T1=350K) using only the Pi group mapping — this is the acid test for dimensional consistency.

12. Transfer Learning Strategies for Engineering Surrogates

Transfer learning reuses knowledge from a source domain (one engine variant, one material, one operating condition set) to a target domain with limited new data. Three strategies in order of interpretability and data efficiency:

12.1 KO-Style Transfer (Source as LF, Target as HF)

Treat the source domain surrogate as the LF model and target domain data as HF observations. Apply the KO auto-regressive correction:

$$y_{target}(\mathbf{x}) = \rho\, y_{source}(\mathbf{x}) + \delta(\mathbf{x}), \quad \delta \sim \mathcal{GP}(0, k_\delta)$$

Requires only $5(d+1)$ target domain points. $\rho$ quantifies how well the source model generalises to the target. This is the most interpretable and data-efficient transfer strategy for engineering surrogates — preferred over NN fine-tuning for small target datasets.

12.2 Fine-Tuning Neural Networks

Freeze the feature extractor layers (backbone); retrain only the output head on target domain data. Effective when source and target share low-level physics (same combustion mechanism, different injection geometry). Requirements:

  • Source and target inputs are in the same space (same signal types, similar operating range)
  • Target dataset: at least 50–100 samples for stable fine-tuning
  • Use a much lower learning rate (10–100× lower) for fine-tuning than initial training — prevents catastrophic forgetting
import torch

def finetune_surrogate(source_model, X_target, y_target, n_epochs=500):
    """
    Fine-tune a source domain surrogate on target domain data.
    Freezes backbone; only output head is updated.
    """
    # Freeze all layers except the last
    for name, param in source_model.named_parameters():
        param.requires_grad = 'output' in name or 'head' in name

    # Fine-tune with 10x lower LR
    opt = torch.optim.Adam(
        filter(lambda p: p.requires_grad, source_model.parameters()),
        lr=1e-4  # 10x lower than initial training
    )
    for epoch in range(n_epochs):
        opt.zero_grad()
        pred = source_model(X_target)
        loss = torch.nn.functional.mse_loss(pred, y_target)
        loss.backward(); opt.step()
    return source_model

12.3 Domain Adaptation with MMD

When the input distribution shifts significantly between source and target (different test conditions, measurement setup), align the feature distributions using Maximum Mean Discrepancy (MMD) as a regularisation term during training:

$$\mathcal{L}_{total} = \mathcal{L}_{task}(\mathbf{X}_{src}, y_{src}) + \lambda\, MMD^2(\phi(\mathbf{X}_{src}), \phi(\mathbf{X}_{tgt}))$$ $$MMD^2 = \left\|\frac{1}{n_s}\sum_i\phi(\mathbf{x}^s_i) - \frac{1}{n_t}\sum_j\phi(\mathbf{x}^t_j)\right\|_\mathcal{H}^2$$

where $\phi$ are the penultimate-layer feature embeddings and the MMD is computed in the RKHS of an RBF kernel.

Transfer Learning Decision Hierarchy
  1. Try KO-style transfer first: interpretable, data-efficient, no retraining.
  2. If KO gives $\rho < 0.8$: source and target are too different for simple correction. Consider fine-tuning.
  3. Fine-tune if target $n \geq 50$: freeze backbone, retrain head.
  4. Apply MMD if the input distributions differ significantly: covariate shift between source and target test conditions.
  5. If target $n < 20$: use KO regardless — NN fine-tuning on 20 points produces unreliable results.

14. Key Takeaways

  • Kennedy-O'Hagan: $y_{HF} = \rho y_{LF} + \delta$. Just 15–30 HF points correct a 200-point LF surrogate. Check $\rho$: if $\rho < 0.8$, fix the LF physics first, don't throw more HF data at it.
  • Budget allocation: if $c_{HF}/c_{LF} > 10$, invest 80% in LF. Minimum HF: $5(d+1)$ for the discrepancy GP to be identifiable.
  • Physics constraints are non-negotiable. Use architecture constraints (zero cost, hard) when the physical relationship is known a priori. Use grey-box (physics + ML residual) as the default for regulated domains.
  • PINNs have narrow applicability: forward problems with known smooth PDEs and sparse data. They fail for turbulence, shocks, and stiff systems. Know the failure modes before committing to PINNs.
  • SINDy for equation discovery: when you want interpretable ODEs, not a black box. Requires low-noise data and a well-chosen candidate function library.
  • POD-ROM: 20–50 modes capture 99% of variance in most engineering flows. POD-surrogate pipeline makes full-field CFD/FEM predictions tractable for design optimisation with existing solver ensembles.
  • Always apply Buckingham Pi before feature engineering. Dimensionless groups reduce input dimension, guarantee scale invariance, and make the surrogate physically interpretable.
  • Transfer learning: KO-style transfer (source as LF) is the most interpretable and data-efficient strategy. Fine-tune NNs only when target dataset is large enough (≥50 points).
