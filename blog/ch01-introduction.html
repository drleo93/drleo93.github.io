Part I — The Engineering Data Foundation · Chapter 1

Introduction: The Engineering Case for Machine Learning, Problem Framing, and the Baseline Discipline

"A problem well-stated is a problem half-solved. A baseline well-chosen is a project half-completed." — after Kettering & Deming

problem framing surrogate models baseline discipline ODD / OOD value of information generative AI limits economics of ML
TL;DR — Five-Minute Version
  • Engineering ML differs from generic ML in three irreducible ways: physical laws are hard constraints (not soft regularisation), data is expensive (not free from the internet), and deployment is the most constrained step (not an afterthought).
  • Before touching any data, answer eight framing questions. The most common project failure is not a bad algorithm — it is a badly framed problem. Framing takes one day; bad framing costs three months.
  • Define and build the baseline model before the complex model. A sophisticated surrogate that does not convincingly beat a default XGBoost on held-out GroupKFold data is solving the wrong problem.
  • Generative AI can accelerate boilerplate and documentation, but cannot reason about your data's physical structure, detect ODD violations, or be accountable for a safety-critical deployment decision. The fundamentals are not optional.

Why Engineering ML Is Not Generic ML

Consider two scenarios. A propulsion engineer fits a Gaussian Process surrogate to 47 LOX/methane hot-fire tests (€25,000–80,000 each) and uses Bayesian optimisation to find the optimal mixture ratio in 30 more tests — cutting a 256-run full-factorial campaign by 70%, from three years to three weeks. A second engineer trains a 5-layer LSTM on an 80/20 random split of test-bench data, achieves CV RMSE = 7.2%, deploys it in a health management system, and watches it trigger false alarms on throttle transients — because the random split put correlated runs in both train and test sets (temporal leakage), and because the model was never exercised on the throttle-down regime. The first engineer applied engineering methodology. The second applied generic ML methodology. The difference is not algorithmic — it is methodological.

Three properties of engineering problems make every step of the ML workflow fundamentally different:

The Three Irreducible Differences
  1. Physical laws are hard constraints, not soft regularisation. Mass is conserved. Entropy increases. Isentropic efficiency cannot exceed unity. A neural network predicting negative species concentrations or specific impulse above the thermodynamic ceiling is not merely inaccurate — it is physically meaningless and may drive a control system to a dangerous state.
  2. Data is precious, expensive, and physically structured. A single RANS CFD simulation of a combustion chamber takes 6–48 hours on a 128-core cluster. A test-bench point costs €500–2,000. There is no ImageNet equivalent for reactor kinetics data. Every point must be planned, justified, and documented.
  3. Deployment constraints are non-negotiable. A surrogate that cannot run in a vehicle ECU within 1 ms with 64 KB of memory, or pass EASA Learning Assurance Level review, or be exported as an FMU compatible with GT-Power, is irrelevant regardless of its RMSE.

The Four Surrogate Model Families

Every engineering surrogate belongs to one of four families. Choosing the wrong family wastes months. The decision is governed by sample size \(n\), input dimension \(d\), and whether calibrated uncertainty is a hard requirement.

Family Uncertainty Interpretability Min. data Best for
RSM (polynomial) None Highest — explicit equations \(3p\) points Regulatory docs, \(d \le 6\)
GP / Kriging Exact analytic Medium \(10d\) points UQ, active learning, DOE
Tree ensembles (XGBoost) Partial (conformal) Medium \(50d\) points Tabular data, physical constraints
Deep learning (MLP/LSTM) Partial (ensemble) Low \(500d\) points or \(n > 5000\) Time series, images, \(n > 5000\)

The model selection decision tree reads as follows: if \(n < 200\) and \(d \le 6\) with a smooth response, use RSM or GP. If calibrated uncertainty is required regardless of \(n\), use a GP (sparse for \(n > 5000\)). For data-rich tabular problems, start with XGBoost. For time series and image data, use LSTM/TCN or MLP. Critically: consult this chart only after the baseline cascade (§ Baseline Protocol below) has been run.

Model Selection Rule of Thumb

If GP beats Default XGBoost by <5% RMSE and you have no UQ requirement, use XGBoost. It is faster to train, faster to deploy, and requires no hyperparameter tuning.

Engineering ML Economics and the Value-of-Information Framework

The economic question for a data acquisition decision is not "how much does this test cost?" but "how much variance in the surrogate prediction does this test eliminate?" This is the Value of Information (VoI) framework.

\[ \text{VoI}(\mathbf{x}^*) = \underbrace{\sigma^2_{\text{prior}}(\mathbf{x}_{\text{target}})}_{\text{current variance}} - \mathbb{E}_{p(y^* \mid \mathbf{x}^*, \mathcal{D})}\!\left[\sigma^2_{\text{post}}(\mathbf{x}_{\text{target}}) \mid y(\mathbf{x}^*)\right] \]

For a GP surrogate, the expected posterior variance reduction from one new observation at \(\mathbf{x}^*\) is analytic:

\[ \Delta\sigma^2(\mathbf{x}_{\text{target}}) = \frac{k(\mathbf{x}_{\text{target}},\,\mathbf{x}^*)^2}{k(\mathbf{x}^*,\,\mathbf{x}^*) + \sigma_n^2} \]

Compare this quantity to the test cost to decide whether the experiment is economically justified. The Expected Improvement acquisition function in Bayesian optimisation is a special case of VoI where the target is the global optimum rather than a specific prediction point.

Example — ROI Calculation for a Hot-Fire Test Campaign

A LOX/methane surrogate has current prediction uncertainty \(\sigma = 4.2\,\text{s}\) on \(I_{sp}\). Qualification requires \(\sigma \le 1.5\,\text{s}\).

  • Option A: 5 hot-fire tests (Sobol' design, €200,000). Expected \(\sigma = 1.1\,\text{s}\). Qualification passed.
  • Option B: 2 CFD runs (€5,000). Expected \(\sigma = 1.8\,\text{s}\). Misses target.
  • Option C: 2 CFD + 2 hot-fire (€120,000). Expected \(\sigma = 1.3\,\text{s}\). Qualification passed at 60% of Option A's cost.

The marginal value of the 3 additional hot-fire tests in Option A is €80,000 for 0.2 s of additional precision — below the qualification threshold anyway. Option C is the correct engineering decision.

When ML Is Not the Right Tool

The economic analysis sometimes returns the right answer: do not build an ML model. ML is wrong when:

  • The physics model already meets the accuracy requirement. A validated Zeldovich NOx correlation at ±8% beats an LSTM at ±7% — the LSTM adds complexity without meaningful benefit.
  • The data acquisition cost exceeds the value of the improved prediction.
  • The deployment environment prohibits ML (e.g., IEC 61508 SIL-3 nuclear safety systems).
  • Explainability is legally required and the model cannot provide it.
The ML Go/No-Go Checklist
  1. Does the best existing physics model meet the accuracy requirement? If yes: use it.
  2. Is the cost of sufficient data less than the value of the improved prediction? If no: use GP, RSM, or a calibrated correlation.
  3. Can the deployment environment run the required model class? If no: define the constraint first, select the model second.
  4. Can the model be validated to the required standard? If no: the project is regulatory-limited, not ML-limited.

The Eight Problem Framing Questions

Answer these questions in writing before opening a Python notebook. Share them with your team. Revisit them when the project hits a wall. The most common engineering ML failure mode is not a bad algorithm — it is a badly framed problem that took three months to reveal itself.

  1. What decision does the model serve? A reformer conversion surrogate for offline optimisation has completely different accuracy, latency, and uncertainty requirements than the same surrogate used in real-time process control.
  2. What is the target variable, in what units, and what is its physically meaningful range? "NOx" is insufficient. "Tailpipe NOx in g/km, expected range 0.01–2.0 g/km, regulatory limit 0.060 g/km under Euro VII" is a proper specification.
  3. What does success look like, quantitatively? State the metric, the target value, and why that target was chosen. "MAPE < 7% on WLTC because the OBD tolerance budget is 10% and we allocate 3% to sensor uncertainty" is a proper success criterion.
  4. What data is available, and what is the marginal cost of one more point? Inventory size, provenance, sampling rate, calibration certificates, missing fraction, and cost of the next experiment.
  5. What does the simplest possible model look like? Linear regression? A lookup table? The Zeldovich correlation? Define the baseline before defining the model.
  6. What physical constraints must the model respect? Monotonicity requirements, conservation laws, bounds, stability requirements. These constrain architecture choices.
  7. Where will the model run, and what are the resource constraints? A model on a cloud server and one in a FADEC computer with 256 KB RAM are architecturally incompatible. Know the deployment target before training.
  8. How will the model be validated, and to what standard? ASME V&V 10? EASA CM-0064? ISO 26262 ASIL-B? Know the regulatory requirements before the project starts, not after.

The Baseline Protocol: A Non-Negotiable Discipline

The baseline model is not a warm-up exercise. It is the hypothesis test that your complex model must pass. No surrogate leaves the lab without convincingly beating the baseline cascade on the correct GroupKFold split.

The Four-Step Baseline Cascade (Regression)
  1. Naïve: predict the training mean \(\hat{y} = \bar{y}\). RMSE\(_\text{naïve} = s_y\). Any model with higher RMSE has negative skill and must be discarded.
  2. Linear OLS: all features. If \(R^2 > 0.99\): stop here. It is interpretable, deployable, and certifiable.
  3. RSM / lookup table: second-order polynomial or interpolated table. This is the industry standard in powertrain calibration and chemical process control.
  4. Default XGBoost: shallow trees at library defaults, no hyperparameter tuning, GroupKFold CV with your physical groups. Your GP or LSTM must beat this before it is considered.

If your complex model does not beat Default XGBoost by >20% RMSE on the correct GroupKFold split, one of three things is true: (a) the problem is simple — use XGBoost; (b) your complex model has a data leak; (c) the data does not contain enough information. All three discoveries are valuable.

Defining the Operational Design Domain (ODD)

The Operational Design Domain is the set of input conditions within which the model is asserted to produce reliable predictions. Defining it correctly is as important as training the model. Under EASA CP-003 (2023) and ISO 26262, silent extrapolation beyond the ODD is a regulatory violation.

ODD Requirements (EASA-aligned)
  1. Explicit: every input variable has stated minimum and maximum bounds.
  2. Tested: a validation dataset covers the ODD boundaries.
  3. Enforced: at inference time, inputs outside the ODD trigger a defined fallback — conservative value, alarm, or explicit OOD flag. Never silent extrapolation.
ODD Definition Checklist
  1. List all input variables with physical min/max (hard bounds from physics or hardware).
  2. List the operational bounds — the subset representing normal operation (narrower than physical bounds).
  3. Identify nonlinear feasibility constraints between inputs (e.g., high \(T\) + low \(S/C\) causes carbon deposition — infeasible even if each is individually within bounds).
  4. At inference: hard-reject inputs outside physical bounds; warn on inputs outside operational bounds.
  5. Re-validate when: a new operating regime is requested, a hardware change occurs, or the input distribution shifts >10% in any direction (detected by CUSUM monitoring).

On Generative AI in Engineering ML

Pitfall — The Most Expensive Shortcut

An LLM told an engineer that GroupKFold was "an optional refinement for time series." It is not. It is the correct cross-validation strategy for any dataset with physical measurement groups, and ignoring it produces optimistic RMSE estimates that collapse on deployment. The LLM was confident, fluent, and wrong — because it was producing statistically likely text about cross-validation, not reasoning about hot-fire test data from a rocket engine campaign.

A language model produces output by completing patterns observed in its training corpus. It is systematically unreliable for tasks where the correct answer requires:

  • Reasoning about your specific data. A model can say "GPs generalise well with small datasets." It cannot say whether your 47 hot-fire points adequately cover your engine's operational envelope at your throttle schedule. The engineer who knows Sobol' sensitivity analysis can answer that from the data. The engineer who asked an LLM cannot.
  • Identifying when physics constraints are violated. A surrogate predicting \(I_{sp}\) above the thermodynamic ceiling for LOX/methane is physically impossible. An LLM will not flag this without very specific prompting — and may hallucinate a plausible-sounding but incorrect thermodynamic justification.
  • Distinguishing data leakage from clean validation. Random KFold looks syntactically identical to GroupKFold. The difference is conceptual: GroupKFold is correct because test-bench runs share temporal correlation. Understanding why requires knowing the physical origin of the data.
  • Making safety-critical decisions under uncertainty. The decision to deploy a virtual sensor for turbopump health monitoring requires calibrated uncertainty, domain knowledge, and accountability. No language model can be accountable for a propulsion system.

Best Practice — Safe LLM Use in Engineering ML
  1. Verify everything in the "Requires verification" tier against a primary source before using it.
  2. Never delegate "Unreliable" tier decisions to an LLM without independent expert review.
  3. Use LLMs to generate options, not to select them. "What CV strategies exist?" is safe. "Which CV strategy should I use for my data?" is not.
  4. Treat LLM output as a first draft, not a final answer. The speed gain is real. The intellectual shortcut is an illusion with a deferred cost.

Key Takeaways

Chapter Summary
  • Engineering ML fails on three fronts generic ML does not face: violated physics, expensive data, and demanding deployment constraints. Every chapter in this book addresses all three.
  • Problem framing precedes data collection. The eight framing questions prevent the most common and expensive project failures.
  • The baseline protocol is not optional. Define the four-step cascade before training any complex model; beat it convincingly on GroupKFold before deploying.
  • The Operational Design Domain is a regulatory deliverable under EASA and ISO 26262. Silent extrapolation outside the ODD is a safety violation, not a modelling choice.
  • Data-first discipline: weak correlations between physical variables are often the most important ones. Examine the data in physics-motivated coordinate systems before running any model. Observe first, hypothesise second, model third.
  • The methods in this book connect into a Digital Twin architecture: surrogate modelling + system identification + virtual sensors + condition monitoring + optimisation + reliability. The Digital Twin chapter maps the full lifecycle and shows how every chapter in the book contributes.
