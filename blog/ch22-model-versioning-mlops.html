Model Versioning, Lifecycle Management, and MLOps

From experiment tracking to certification evidence — the infrastructure that makes engineering ML auditable and reproducible.
MLflowDVCModel Registry PSIRollbackDO-178C ISO 26262ONNX
TL;DR — A model that cannot be reproduced, audited, or rolled back is a liability. Every deployed engineering model needs a unique version linking training-data hash, hyperparameters, validation results, and the code that produced it. Use MLflow + DVC for tracking, a model registry for lifecycle management, PSI for drift detection, and maintain an immutable certification evidence package for regulated applications.

Why This Matters

Software version control (Git) tracks code. Model versioning tracks the combination of code, data, configuration, and environment that produced a specific artefact. The same code run on a different dataset version, or a different random seed, produces a different model — and that model may be deployed in safety-critical hardware.

Pitfall — "It worked on my machine"
An ML engineer trains a GP surrogate, achieves CV RMSE = 0.63%, and sends model.pkl to the ECU integration team. Six months later, a safety audit requests: (1) the training data used, (2) the exact hyperparameters, (3) the validation results, (4) proof that the deployed model matches the validated one.

Without version control: none of this can be recovered. The audit fails. The certification timeline slips 4 months.
With a model registry: all four items retrieved in 5 minutes.

Experiment Tracking with MLflow

MLflow is the industry standard for engineering ML experiment tracking. Every training run logs parameters, metrics, artefacts, and tags — forming a reproducible record of every model ever trained. 

import mlflow, mlflow.sklearn
import numpy as np
from sklearn.gaussian_process import GaussianProcessRegressor
from sklearn.gaussian_process.kernels import Matern, WhiteKernel

mlflow.set_experiment("LOX_CH4_Isp_surrogate")

with mlflow.start_run(run_name="GP_Matern52_ARD_v3"):
    # Log parameters
    mlflow.log_params({
        "kernel":         "Matern52_ARD + WhiteKernel",
        "n_restarts":     5,
        "normalize_y":    True,

        "n_train":        47,

    })

    # Train

                                  normalize_y=True)

Data Versioning with DVC

Use DVC (Data Version Control) alongside Git to version datasets. dvc add data/campaign_2024.csv creates a .dvc pointer file that Git tracks; the actual data lives in a remote store (S3, Azure Blob, NAS). Every training run records which DVC-tracked dataset version it used — enabling exact reproduction.

Four artefacts to version for every model
  1. Git commit hash — the exact code version
  2. DVC dataset hash — the exact training data
  3. MLflow run ID — the hyperparameter configuration and metrics
  4. ONNX model SHA-256 — the deployed binary
All four must be linked in the model card. A certification audit that cannot verify all four fails.

The Model Registry

The model registry is a versioned catalogue of model artefacts with lifecycle state. MLflow Model Registry defines four stages — each with mandatory engineering exit criteria.

Semantic Versioning for Models

Adopt MAJOR.MINOR.PATCH versioning:

  • MAJOR: breaking change to input feature set — old ECU code cannot call the new model without update.
  • MINOR: new training data, improved accuracy, same feature set.
  • PATCH: hyperparameter refinement, no accuracy change, same data and features.

The Model Lifecycle: Development to Retirement

The engineering model lifecycle has four phases. Each is separated by a mandatory quality gate. Failed gates return to Development — not to the previous gate.

Lifecycle phase flow
Development → [CV metrics met?] → Validation (SIL + HIL) → [HIL WCET passed?] → Shadow Mode → [Shadow delta OK?] → Production → [Retire] → Archived

Failure at any gate loops back to Development, not just the previous phase — ensuring the entire validation chain is re-executed.

Continuous Monitoring and Retraining Triggers

Four signals trigger retraining or rollback in production:

  1. Data drift (covariate shift): input distribution \(p(\mathbf{x})\) has changed. Detect with Population Stability Index (PSI) or Maximum Mean Discrepancy (MMD). PSI > 0.20: trigger retraining.
  2. Prediction drift: output distribution \(p(\hat{y})\) shifts without input change — may indicate physical process change (catalyst degradation).
  3. Performance degradation: RMSE on labelled production samples exceeds \(1.5\times\) the deployment-time RMSE.
  4. ODD violation rate: fraction of production queries flagged as outside the ODD exceeds 5% over a rolling 24-hour window.

Population Stability Index (PSI)

import numpy as np

def psi(expected: np.ndarray, actual: np.ndarray, bins=10) -> float:
    """
    Population Stability Index between expected (training) and actual (production).
    PSI < 0.10: no significant change.
    PSI 0.10-0.20: moderate change; monitor closely.
    PSI > 0.20: significant change; retrain.
    """
    eps = 1e-9
    edges = np.percentile(expected, np.linspace(0, 100, bins + 1))
    edges[0] -= eps; edges[-1] += eps
    p_exp = np.histogram(expected, bins=edges)[0] / len(expected) + eps
    p_act = np.histogram(actual,   bins=edges)[0] / len(actual)   + eps
    p_exp /= p_exp.sum(); p_act /= p_act.sum()
    return float(np.sum((p_act - p_exp) * np.log(p_act / p_exp)))
$$\text{PSI} = \sum_{i=1}^{B} (p_{act,i} - p_{exp,i}) \ln\!\frac{p_{act,i}}{p_{exp,i}}$$

Rollback Protocol

A rollback is the immediate reactivation of the previous production model version. It is not a failure — it is a designed safety mechanism.

Rollback decision criteria
  1. RMSE on labelled production samples > 2× deployment-time RMSE.
  2. ODD violation rate > 10% over 1 hour.
  3. Any safety-critical prediction differs from the previous model version by more than the pre-specified tolerance.
  4. Automated rollback within one OPC-UA update cycle (≤ 100 ms): the model registry must store the ONNX binary of the previous version; the ECU must maintain both in memory during any A/B deployment.

Shadow → Canary → Champion-Challenger Rollout

Never skip shadow mode for safety-critical models. The three-stage rollout:

Shadow mode0% (inference only, no actuation)Validate production predictions against champion without risk4-week delta statistics within tolerance
Canary5–10%Real-world performance on limited trafficNo safety alerts; RMSE equal or better than champion
Champion-Challenger50/50Statistical significance test before full switchoverChallenger RMSE significantly better; rollback triggered if worse

Certification Evidence Package

For safety-critical deployment (DO-178C, ISO 26262, EASA CP-003), the certification evidence package for model version \(v\) must contain:

  1. Version traceability: Git commit hash + DVC dataset hash + MLflow run ID + ONNX SHA-256. All four linked in the model card.
  2. Validation evidence: signed 15-step validation report run on the exact registered model artefact.
  3. HIL/SIL evidence: WCET measurement log from target hardware with pass/fail against timing budget.
  4. Shadow mode evidence: 4-week shadow mode log showing prediction delta statistics vs. current production model.
  5. Monitoring plan: PSI threshold, retraining trigger, and rollback criteria in force for this version.
  6. Retirement date: maximum calendar date or accumulated operating hours after which this version must be retrained or retired, regardless of monitoring alarms.
Immutable model artefacts
Once a model transitions to Staging or above, its artefact (ONNX binary + metadata) must be immutable. Overwriting a registered model is equivalent to destroying certification evidence. Use append-only storage (S3 with object lock, Azure Immutable Blob Storage) for the model registry backend in regulated applications.

Key Takeaways

  • Model versioning tracks code + data + config + environment — not just code. Use MLflow + DVC.
  • The four lifecycle stages (None → Staging → Production → Archived) each have mandatory engineering exit criteria.
  • PSI > 0.20 triggers retraining. RMSE > 2× triggers rollback. Both thresholds must be pre-specified before deployment.
  • Shadow mode is mandatory before any A/B exposure for safety-critical models.
  • Certification evidence = Git hash + DVC hash + MLflow run ID + ONNX SHA-256. All four. Always.
