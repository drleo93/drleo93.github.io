Virtual Sensors: From Concept to Deployment

Chapter 12 — Design, Grey-Box NOx, EKF Fusion, CUSUM Monitoring
Virtual Sensors Grey-Box NOx EKF CUSUM ONNX OBD Regulatory

TL;DR

  • A virtual sensor estimates a physical quantity from available signals. Design the observability analysis first — if $I(\mathbf{x}; y_v) \approx 0$, no model will help.
  • Grey-box (physics steady-state + ML transient correction) requires 40% less data than pure black-box and is physically consistent — mandatory for regulatory applications.
  • EKF fuses physics-based state-space dynamics with sensor observations; use it for battery core temperature where safety margins matter.
  • CUSUM detects model drift in production with a defined average run length (ARL⊂0; = 370 at $k=0.5$, $h=5$). Always deploy a monitoring chart alongside the model.
  • Regulatory deployment (Euro 7 OBD, EASA LAL-B) requires model cards, calibration traceability, uncertainty budget (GUM), and documented ARL.

1. The 8-Step Virtual Sensor Design Process

Design Process
  1. Target identification: which physical quantity $y_v$ is to be estimated? What accuracy and latency are required?
  2. Observability analysis: can $y_v$ be uniquely determined from available signals $\mathbf{x}$? Check mutual information $I(\mathbf{x}; y_v)$ and physical causality.
  3. Signal selection: select the minimal set $\mathbf{x}^* \subset \mathbf{x}$ using mutual information + Granger causality + domain knowledge.
  4. Architecture selection: 0D static (GP, XGBoost) vs. dynamic (LSTM, TCN, EKF+LSTM).
  5. Calibration: train on labelled dataset; apply nested CV; ensure generalisation to all operating regimes.
  6. Validation: 15-step protocol plus regulatory requirements (MAPE, residual whiteness, simulation error).
  7. Deployment: ONNX export, latency profiling, memory footprint verification.
  8. Production monitoring: CUSUM/EWMA drift detection; scheduled recalibration protocol.

2. Virtual NOx Sensor: A Complete Case Study

Physical Background

NOx in diesel combustion follows the thermal Zeldovich mechanism. The rate constant $k_1 = 1.8\times 10^{11} \exp(-38{,}370/T)$ makes NOx formation an exponential function of peak combustion temperature. The 7 ECU inputs that physically drive NOx:

SignalSymbolPhysical Role
Engine speed$N$ [rpm]Residence time and charge motion
Fuel quantity$q_f$ [mg/cyc]Chemical energy input
Injection timing$\phi_{SOI}$ [°CA]Heat release phasing
Rail pressure$p_{rail}$ [bar]Mixture preparation quality
EGR rate$x_{EGR}$ [%]Directly reduces $T_{peak}$
Boost pressure$p_{boost}$ [bar]Charge density
Air-fuel ratio$\lambda$ [-]Excess oxygen for NOx formation

Regulatory context: EU 715/2007 (Euro 6d) mandates on-board NOx monitoring. Virtual NOx sensor must achieve MAPE <10% on the WLTC to satisfy OBD requirements (no physical PEMS analyser needed, saving ~50,000 €).

Grey-Box Architecture

The virtual sensor combines a GP steady-state component with an LSTM transient correction:

$$\hat{y}_{\text{NOx}}(t) = \underbrace{\hat{f}_{\text{GP}}(\mathbf{x}_{SS}(t))}_{\text{steady-state map}} + \underbrace{\hat{\delta}_{\text{LSTM}}(\mathbf{x}(t-W:t))}_{\text{transient correction}}$$ 
class GreyBoxNOxSensor(nn.Module):
    """Grey-box virtual NOx sensor: GP steady-state + LSTM transient correction."""
    def __init__(self, n_ss_features=7, n_ts_features=10, window=50, lstm_hidden=64):
        super().__init__()
        self.lstm = nn.LSTM(n_ts_features, lstm_hidden,
                            num_layers=2, batch_first=True, dropout=0.1)
        self.delta_head = nn.Sequential(
            nn.Linear(lstm_hidden, 32), nn.GELU(), nn.Linear(32, 1)
        )

    def forward(self, x_window, gp_ss):
        out, _ = self.lstm(x_window)
        delta = self.delta_head(out[:, -1, :]).squeeze(-1)
        return gp_ss + delta

# Training protocol:
# 1. Train GP on steady-state (transient-removed) data
# 2. Compute GP predictions on full dataset
# 3. Train LSTM on residuals: y_true - y_GP
# Note: LSTM only learns what the physics GP cannot explain
Lesson
Always log-transform NOx before GP fitting: gpflow.models.GPR(data=(X_norm, np.log1p(y))). NOx is log-normally distributed — a GP on raw NOx values will underperform significantly. Back-transform with np.expm1().

4. Extended Kalman Filter for Battery Temperature

The EKF handles nonlinear state-space models by linearising at the current estimate. For battery core temperature (state $\mathbf{x} = [T_{core}, T_{surf}]^\top$):

EKF Update Step: $$\hat{\mathbf{x}}_{t|t} = \hat{\mathbf{x}}_{t|t-1} + \mathbf{K}_t\bigl(y_t - h(\hat{\mathbf{x}}_{t|t-1})\bigr)$$ $$\mathbf{K}_t = \mathbf{P}_{t|t-1}\mathbf{H}_t^\top(\mathbf{H}_t\mathbf{P}_{t|t-1}\mathbf{H}_t^\top + \mathbf{R})^{-1}$$ Lumped thermal model (Prediction Step): $$\dot{T}_{core} = \frac{I^2 R_0}{C_c} - \frac{T_{core} - T_{surf}}{R_{cc} C_c}$$ $$\dot{T}_{surf} = \frac{T_{core} - T_{surf}}{R_{cc} C_s} - \frac{T_{surf} - T_{amb}}{R_{su} C_s}$$

The EKF observation model: $y = T_{surf,meas}$ (only surface temperature is measured). The EKF fuses this observation with the physics-based prediction to estimate the unobservable core temperature — critical for thermal runaway prevention.

Lesson
Use EKF (not a pure neural network) for battery core temperature. The physical thermal model provides hard constraints on heat flow — a black-box NN may predict physically impossible temperature gradients that lead to incorrect safety decisions. The Kalman gain $\mathbf{K}_t$ automatically weights model vs. measurement uncertainty.

5. Production Monitoring with CUSUM

The CUSUM chart accumulates evidence for a mean shift $\delta$ in the normalised model residuals $\{e_t\}$:

$$C_t^+ = \max(0,\, C_{t-1}^+ + e_t - k) \quad \text{(upper — positive shift)}$$ $$C_t^- = \max(0,\, C_{t-1}^- - e_t - k) \quad \text{(lower — negative shift)}$$ $$\text{Alarm when } C_t^+ > h \text{ or } C_t^- > h$$

Standard parameters: $k = \delta/2$ (reference value), $h = 5$ → ARL$_0 \approx 370$ (on average 370 samples before a false alarm under in-control conditions).

from dataclasses import dataclass, field

@dataclass
class CUSUMMonitor:
    """CUSUM control chart for virtual sensor drift detection."""
    k: float = 0.5      # reference value (delta/2 of shift to detect)
    h: float = 5.0      # decision threshold (ARL~370 for standard normal)
    C_pos: float = field(default=0.0, init=False)
    C_neg: float = field(default=0.0, init=False)

    def update(self, residual: float) -> bool:
        """Update with normalised residual. Returns True if alarm."""
        self.C_pos = max(0.0, self.C_pos + residual - self.k)
        self.C_neg = max(0.0, self.C_neg - residual - self.k)
        return (self.C_pos > self.h) or (self.C_neg > self.h)

    def reset(self):
        self.C_pos = self.C_neg = 0.0

# Usage in production pipeline
monitor = CUSUMMonitor(k=0.5, h=5.0)
for i, (y_true, y_pred, std_dev) in enumerate(production_stream):
    norm_residual = (y_true - y_pred) / std_dev   # standardise by model sigma
    if monitor.update(norm_residual):
        trigger_recalibration_alert(sensor_id='NOx_VS_01', cycle=i)
        monitor.reset()

6. Regulatory Deployment Checklist

Regulatory Deployment Requirements
  1. Model card: document training data provenance, preprocessing pipeline (with scaler state), hyperparameters, validation results, and known operating range.
  2. Calibration traceability: every training run is version-controlled (DVC). Model hash stored in ECU software manifest.
  3. Uncertainty budget: specify combined uncertainty $u_c$ following GUM. For Euro 7 OBD: $u_c < 5\%$ required.
  4. ASIL decomposition: if the virtual sensor feeds a safety function, decompose ISO 26262 ASIL across the sensor chain; document the QM contribution of the ML model.
  5. Monitoring architecture: deploy CUSUM with documented ARL$_0 \geq 370$. Log all alarm events. Define recalibration triggers upfront.

7. Key Takeaways

  • The 8-step process is not optional — skipping observability analysis or signal selection leads to a model that cannot generalise.
  • Grey-box = physics steady-state + ML transient correction. Almost always the right architecture for regulated engineering domains.
  • EKF for safety-critical state estimation (battery temperature, SoC): physical model provides hard constraints; Kalman gain weights model vs. observation uncertainty.
  • CUSUM is the production monitoring standard: $k=0.5$, $h=5$, ARL$_0 \approx 370$. Normalise residuals by model predictive uncertainty $\sigma$.
  • For Euro 7/EASA certification: model card + traceability + GUM uncertainty budget + documented ARL are all mandatory, not optional.
  • ONNX export is the standard for cross-platform deployment. Always profile latency on the target hardware, not the development machine.
