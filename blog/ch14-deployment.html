2. ONNX: The Universal Export Pipeline

ONNX (Open Neural Network Exchange) is the universal intermediate representation. Train in any Python framework, export once, run anywhere:

  • XGBoost/sklearn → skl2onnx → ONNX
  • PyTorch → torch.onnx.export() → ONNX
  • TensorFlow → tf2onnx → ONNX
  • GPflow → TF → ONNX

ONNX Runtime executes on any target: Linux, Windows, ARM MCU (via TFLite), GPU (via TensorRT), or bare-metal ECU (via ONNX2C C-code generation).

# Export sklearn/XGBoost pipeline to ONNX
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
import onnxruntime as ort
import numpy as np

initial_type = [('float_input', FloatTensorType([None, d]))]
onnx_model = convert_sklearn(pipeline, initial_types=initial_type, opset=17)

with open("surrogate.onnx", "wb") as f:
    f.write(onnx_model.SerializeToString())

# Validate numerical equivalence
sess = ort.InferenceSession("surrogate.onnx")
y_python = pipeline.predict(X_test)
y_onnx   = sess.run(None, {'float_input': X_test.astype(np.float32)})[0]
max_rel_error = np.max(np.abs(y_python - y_onnx) / (np.abs(y_python) + 1e-10))
assert max_rel_error < 1e-4, f"ONNX equivalence FAIL: {max_rel_error:.2e}"

# Latency benchmark
import time
latencies = []
for _ in range(1000):
    t0 = time.perf_counter()
    sess.run(None, {'float_input': X_test[0:1].astype(np.float32)})
    latencies.append((time.perf_counter() - t0) * 1e6)
print(f"Mean: {np.mean(latencies):.1f} µs, P99: {np.percentile(latencies, 99):.1f} µs")

3. Model Quantisation

INT8 quantisation maps FP32 weights and activations to 8-bit integers:

$$x_{int8} = \operatorname{round}\!\left(\frac{x_{fp32} - z}{s}\right), \quad x_{fp32} \approx s \cdot x_{int8} + z$$

where $s$ (scale) and $z$ (zero-point) are computed from a representative calibration dataset.

FP161×1.5–2×<0.1%LSTM, Transformer, GPU edge

INT40.25× FP164–8×2–10%Ultra-constrained; use carefully
Pitfall — Fixed-Point Overflow at Temperature Extremes
INT8 scale factors are computed from the calibration dataset. If production inputs fall outside the calibration range — which happens predictably at temperature extremes — activations saturate at ±127 and model output is clipped or corrupted. Always calibrate on the full operating envelope, including corner cases (−40°C cold start, +125°C engine bay). Re-run the full validation protocol after quantisation before ECU integration.
Quantisation Rules for Engineering Surrogates
  1. Always re-run the 15-step validation protocol after quantisation.
  2. For regulatory-certified surrogates (FADEC, OBD): document quantisation procedure, calibration dataset provenance, and post-quantisation accuracy in the model card.
  3. INT8 XGBoost: typically <0.5% RMSE increase for smooth regression (compressor map, OCV curve).
  4. LSTM: use dynamic quantisation (weights INT8, activations FP32). Static quantisation of LSTM fails due to hidden state distribution shift.
  5. GP models are FP64 by necessity. For embedded: distil to INT8-quantised MLP or use decision-tree approximation.

4. MLOps: Engineering ML Systems for Production

Only 3% of companies successfully leverage AI to a high degree in production. The gap is engineering infrastructure, not algorithms. Three consistent barriers:

  1. Data accessibility: sensor data exists but is locked in plant historians (OSIsoft PI, AVEVA, AspenTech InSite). Solution: treat the data pipeline as a capital project, not an IT ticket.
  2. Team structure: data scientists and process engineers in separate silos with incompatible KPIs. Solution: cross-functional ML engineering team with both physics knowledge and ML skills.
  3. Wrong approach: attempting to pre-train large models before establishing data infrastructure. Solution: start with a linear model on existing historian data and build infrastructure incrementally.

Model Versioning and Reproducibility

Minimum requirements for production engineering models:

  1. Data versioning: every training dataset has a unique hash. Use DVC or MLflow artifacts to link model versions to the exact data they were trained on.
  2. Code versioning: training code pinned to a Git SHA. The model card records both data hash and code SHA.
  3. Environment reproducibility: requirements.txt frozen to exact package versions. Docker image for complete reproducibility.
  4. Random seed control: all random operations seeded and recorded. Verify: retrain from same data + code + seed; compare model hashes.

A/B Testing and Staged Rollouts

Never replace the production model immediately with a new version:

  • Shadow deployment: new model runs in parallel, predictions logged but not acted on. Compare performance against production on live data.
  • Canary release: route 5–10% of inference requests to the new model. Monitor RMSE and OOD rates. Roll back if metrics degrade.
  • Champion-Challenger: define a statistical test (paired t-test on RMSE) for promotion from Challenger to Champion.

5. Out-of-Distribution Detection

Pitfall — Silent Deployment Failure
The most dangerous deployment failure is silent: a model running outside its training domain, returning confidently wrong predictions with no alarm. Input range validation and OOD detection must be part of every deployed surrogate — not optional.

OOD detection strategies for engineering surrogates:

  • Input range validation: enforce hard bounds on each input feature. If any input falls outside $[\mu_{train} \pm 3\sigma_{train}]$, flag as OOD and return NaN rather than a prediction.
  • Mahalanobis distance: $D^2 = (\mathbf{x} - \boldsymbol{\mu})^\top \Sigma^{-1} (\mathbf{x} - \boldsymbol{\mu})$ computed in the training feature space. Threshold at 99th percentile of training $D^2$ values.
  • GP posterior variance: if using a GP surrogate, the posterior variance $\sigma^{*2}(\mathbf{x})$ directly quantifies prediction uncertainty. Set a maximum variance threshold above which predictions are flagged.
  • Ensemble disagreement: train $M=5$ models on bootstrap samples. Flag if standard deviation of predictions exceeds a threshold.

6. Key Takeaways

  • ONNX is the deployment standard: export once, run anywhere. Always validate numerical equivalence after export (relative error <$10^{-4}$).
  • INT8 gives 4× compression at <2% RMSE loss. Use dynamic quantisation for LSTM. Always calibrate on the full operating envelope.
  • Profile latency on target hardware. Development machine latency is irrelevant for ECU or MCU deployment.
  • MLOps is infrastructure engineering, not a data science task. Data pipeline + versioning + monitoring must be built before deploying models at scale.
  • Shadow deployment → canary release → champion-challenger: always stage model updates in production.
  • OOD detection is non-negotiable. Silent failures from domain shift are more dangerous than model accuracy limitations.
  • For regulatory deployment (FADEC, OBD): model card + traceability (Git SHA + data hash) + post-quantisation validation are all mandatory.
